# 2. 设计原则

## 2.1 单一事实来源原则

以下三类状态分别承担不同职责，不允许混用：

1. **业务对象状态**
    
    - 面向用户、页面和业务流程
    - 如 `TalkingHeadTask.status`
2. **步骤状态**
    
    - 面向模块内的节点执行情况
    - 如 `tts_status`、`compose_status`
3. **Job / JobStep 状态**
    
    - 面向异步执行系统
    - 如 `job.status`、`job_step.status`

任何时候：

- 页面展示主状态，优先读业务对象状态
- 执行细节追踪，读 Job / JobStep
- 某个步骤是否可重新执行，读步骤状态

---

## 2.2 Job 只记录“一次执行尝试”

一次用户点击“运行流水线”，产生一个 Job。  
一次手动重试，**默认产生新的 Job**。  
不建议在同一个 Job 上反复覆盖历史状态。

这意味着：

- Job 天然可审计
- 失败重试不会污染第一次执行记录
- OpenClaw 可明确感知每次尝试

---

## 2.3 Pipeline 优先、子 Job 从简原则

v2.1.1 中：

- Pipeline Job 是一条完整业务链路的执行容器
- 步骤拆到 `job_steps`
- 默认**不再为每一步创建独立子 Job**
- 特殊情况下可扩展，但首版不作为通用模型

这样可避免：

- 父子任务树过深
- 查询链过长
- 状态难以聚合

---

## 2.4 必要步骤与可选步骤分离原则

每个 Pipeline 必须标记：

- **Required Step**：失败会导致本次 pipeline 失败
- **Optional Step**：失败不一定导致 pipeline 失败，可进入 `partial_success` 或 `partial_ready`

---

## 2.5 上游变更传播原则

若上游关键输入发生变化，依赖它的下游步骤必须进入 `stale`。  
`stale` 不等于 `failed`，表示“结果已过期、待重新生成”。

# 3. 状态模型总览

---

## 3.1 三层状态结构

Text

业务对象

└── 业务主状态（draft / ready / processing / partial_ready / blocked / completed / failed / archived）

    └── 模块步骤状态（tts_status / image_status / compose_status ...）

        └── Job 执行状态（queued / running / success / failed ...）

            └── JobStep 执行状态（pending / queued / running / succeeded / failed ...）

---

## 3.2 业务对象状态定义

|状态|含义|
|---|---|
|`draft`|对象已创建但尚未达到可执行条件|
|`ready`|已满足执行条件，可运行|
|`processing`|当前存在进行中的 Job 或等待人工确认的执行链路|
|`partial_ready`|已获得部分可用结果，但仍需补齐或重跑部分步骤|
|`blocked`|存在缺失配置/资产/依赖，无法继续|
|`completed`|本轮业务目标已完成，可导出或已导出|
|`failed`|最近一次执行已失败，且当前无运行中 Job|
|`archived`|已归档，默认只读|

---

## 3.3 步骤状态定义

|状态|含义|
|---|---|
|`pending`|尚未进入执行计划|
|`queued`|已加入执行计划，等待 Worker|
|`running`|正在执行|
|`succeeded`|成功完成|
|`failed`|执行失败|
|`skipped`|本次执行中被跳过|
|`cancelled`|被取消|
|`stale`|结果失效，需要重新生成|

---

## 3.4 Job 状态定义

|状态|含义|
|---|---|
|`queued`|已创建，等待执行|
|`running`|正在执行|
|`partial_success`|部分可选步骤失败，但已有部分可用结果|
|`success`|成功完成|
|`failed`|执行失败|
|`cancelled`|已取消|

---

## 3.5 JobStep 执行附加状态

除 `status` 外，JobStep 增加 `execution_state`：

|execution_state|含义|
|---|---|
|`normal`|正常执行中|
|`waiting_review`|等待人工确认|
|`waiting_callback`|已提交 Provider，等待回调|
|`waiting_polling`|已提交 Provider，等待轮询结果|

> 说明：  
> `waiting_review` / `waiting_callback` / `waiting_polling` 是**执行附加态**，不是替代 `status` 的主状态。  
> 此时 `status` 通常保持 `running`。

# 4. Job 中心详细模型

---

## 4.1 Job 关键字段语义

Ts

interface Job {

  id: string;

  biz_type: string;

  biz_id: string;

  run_type: string;

  trigger_source: 'user' | 'system' | 'openclaw' | 'retry';

  status: 'queued' | 'running' | 'partial_success' | 'success' | 'failed' | 'cancelled';

  current_step?: string;

  priority: number;

  retry_count: number;

  idempotency_key?: string;

  parent_batch_id?: string;

  error_code?: string;

  error_message?: string;

  created_at: string;

  started_at?: string;

  finished_at?: string;

}

### 字段约束

- `biz_type + biz_id`：唯一标识此次执行属于哪个业务对象
- `run_type`：标识执行目标，如 `pipeline`、`tts`、`compose`
- `current_step`：当前执行中的 step_code
- `retry_count`：当前 Job 自动重试次数，不含新建 Job 的手动重试
- `parent_batch_id`：若为批量执行中的单个 Job，则关联批次 ID
- `error_code/error_message`：记录最终失败原因，或最近一次阻断原因

---

## 4.2 JobStep 关键字段语义

Ts

interface JobStep {

  id: string;

  job_id: string;

  step_code: string;

  step_name: string;

  step_order: number;

  required: boolean;

  status: 'pending' | 'queued' | 'running' | 'succeeded' | 'failed' | 'skipped' | 'cancelled';

  execution_state?: 'normal' | 'waiting_review' | 'waiting_callback' | 'waiting_polling';

  provider_name?: string;

  provider_task_id?: string;

  input_snapshot?: Record<string, any>;

  output_snapshot?: Record<string, any>;

  error_code?: string;

  error_message?: string;

  retry_count: number;

  started_at?: string;

  finished_at?: string;

}

### 字段约束

- `step_order`：Pipeline 执行顺序
- `required`：是否必要步骤
- `provider_task_id`：外部异步任务关联键
- `input_snapshot`：执行时输入快照
- `output_snapshot`：执行成功或部分成功后的结果快照
- `retry_count`：该步骤在当前 Job 内的自动重试次数

---

## 4.3 JobEvent 事件规范

### 事件类型建议全集

- `created`
- `queued`
- `started`
- `step_started`
- `provider_submitted`
- `provider_callback_received`
- `provider_polled`
- `waiting_review`
- `review_approved`
- `review_rejected`
- `review_regenerated`
- `retry_scheduled`
- `step_failed`
- `failed`
- `partial_success`
- `completed`
- `cancel_requested`
- `cancelled`
- `restored_after_restart`

### 事件记录规则

1. Job 创建必须记录 `created`
2. Worker 领取必须记录 `started`
3. 每个 step 开始执行必须记录 `step_started`
4. 外部任务提交必须记录 `provider_submitted`
5. 人工确认进入等待态必须记录 `waiting_review`
6. Job 最终结束必须记录 `completed` / `failed` / `cancelled`

# 5. 调度与执行模型

---

## 5.1 总体执行流

Text

用户 / OpenClaw 触发

→ API 参数校验

→ 校验业务对象是否可执行

→ 创建 Job

→ 预生成 JobSteps

→ Job.status = queued

→ Scheduler 扫描并分发

→ Worker 领取 Job

→ Job.status = running

→ 顺序执行 JobStep

   → 同步完成 / 异步回调 / 轮询 / 人工确认

→ 所有必要步骤成功

   → Job.success

   → 回写业务对象 completed / partial_ready / failed

---

## 5.2 Scheduler 扫描策略

### 默认扫描条件

满足以下条件的 Job 可被调度：

1. `status = queued`
2. 未超过全局并发上限
3. 未超过 `run_type` 并发上限
4. 未超过 Provider 并发上限
5. 所属批次未处于 `paused_by_system` 或 `cancelled`
6. 所属业务对象未被归档

---

## 5.3 Worker 领取规则

Worker 每次领取 Job 时必须原子完成：

1. 校验 Job 仍为 `queued`
2. 更新为 `running`
3. 写入 `started_at`
4. 写 `JobEvent.started`

防止重复领取。

---

## 5.4 Step 推进规则

Worker 在 Pipeline Job 中按 `step_order` 顺序寻找**下一可执行步骤**：

### 可执行步骤条件

1. `status = queued`
2. 前置 required 步骤已 `succeeded` 或被允许 `skipped`
3. 当前步骤未处于 `waiting_review`
4. 当前步骤未处于 `waiting_callback`
5. Job 未被取消

### 执行结果分支

- 成功：`succeeded`
- 失败：`failed`
- 跳过：`skipped`
- 进入人工确认：`running + waiting_review`
- 进入外部回调等待：`running + waiting_callback`
- 进入轮询等待：`running + waiting_polling`

---

## 5.5 同步步骤与异步步骤

### 同步步骤

适用于：

- 本地文本处理
- 配置检查
- 本地素材元信息提取
- 本地 FFmpeg 短任务
- 轻量 Agent 输出解析

特点：

- Worker 发起后在当前执行周期内直接拿到结果
- 执行结束时即写 step 最终状态

### 异步步骤

适用于：

- 大模型异步调用
- 图像生成
- 视频生成
- URL 下载
- 长时合成
- 第三方异步 TTS

特点：

- 提交 Provider 后可能立即返回 `provider_task_id`
- step 进入 `running + waiting_callback` 或 `running + waiting_polling`
- Job 保持 `running`

---

## 5.6 Pipeline 结束判定规则

### 成功结束

满足：

- 所有 required step = `succeeded`
- optional step ∈ `succeeded / skipped / failed`
- 且最终可输出结果满足本模块完成条件

→ Job 状态：

- 若 optional step 全部成功/跳过：`success`
- 若存在 optional step failed，但结果仍可用：`partial_success`

### 失败结束

满足任一：

- 任一 required step `failed` 且不可自动恢复
- Job 被取消且无可继续策略
- 人工确认 reject 后未重新进入执行
- 重启恢复后无法继续

→ `failed` 或 `cancelled`

# 6. 状态机详细定义

---

## 6.1 业务状态机

### 通用迁移表

|当前状态|事件|下一状态|备注|
|---|---|---|---|
|`draft`|校验通过|`ready`|必填项齐全|
|`draft`|校验失败|`blocked`|缺关键依赖|
|`blocked`|依赖补齐|`ready`|恢复可执行|
|`ready`|创建并启动 Job|`processing`|进入执行|
|`processing`|必要步骤完成|`completed`|达成目标|
|`processing`|部分结果可用|`partial_ready`|等待人工补齐|
|`processing`|失败且无运行中 Job|`failed`|最近一次失败|
|`partial_ready`|修复后重跑|`processing`|新 Job|
|`partial_ready`|所有必要结果补齐|`completed`|完成|
|`failed`|修复配置但未运行|`ready`|待执行|
|`failed`|手动重试|`processing`|新 Job|
|`completed`|上游变更|`partial_ready`|下游失效|
|`completed`|归档|`archived`|只读|
|`archived`|恢复|`completed/ready`|回归稳定态|

---

## 6.2 步骤状态机

|当前状态|事件|下一状态|
|---|---|---|
|`pending`|预生成并纳入计划|`queued`|
|`queued`|Worker 开始执行|`running`|
|`running`|执行成功|`succeeded`|
|`running`|执行失败|`failed`|
|`running`|用户取消|`cancelled`|
|`queued`|用户取消|`cancelled`|
|`succeeded`|上游变更失效|`stale`|
|`failed`|手动或自动重试|`queued`|
|`cancelled`|手动重跑|`queued`|
|`stale`|重新执行|`queued`|
|`skipped`|条件变化|`queued`|

---

## 6.3 Job 状态机

|当前状态|事件|下一状态|
|---|---|---|
|`queued`|Worker 领取|`running`|
|`queued`|用户取消|`cancelled`|
|`running`|全部 required 成功|`success`|
|`running`|optional 部分失败但结果可用|`partial_success`|
|`running`|required 失败|`failed`|
|`running`|用户取消|`cancelled`|
|`failed`|手动重试|新 Job（旧 Job 保持 failed）|
|`partial_success`|手动补跑|新 Job|
|`cancelled`|手动重试|新 Job|

> 约束：  
> v2.1.1 里，**失败后的继续执行优先通过新建 Job 实现**。  
> 旧 Job 不回写到 `queued` 作为主路径。

---

## 6.4 JobStep 附加执行态机

### waiting_callback

- 进入条件：已提交外部 Provider，等待异步结果
- 退出条件：
    - 收到 webhook 成功 → `succeeded`
    - 轮询成功 → `succeeded`
    - Provider 失败 → `failed`
    - 超时 → `failed`

### waiting_polling

- 进入条件：Provider 无 webhook，仅支持查询
- 退出条件：
    - 轮询完成 → `succeeded`
    - 轮询失败 → `failed`
    - 超时 → `failed`

### waiting_review

- 进入条件：步骤产物需人工确认
- 退出条件：
    - approve → 当前步骤 `succeeded`
    - reject → 当前步骤 `failed` 或指定回退步骤 `queued`
    - regenerate → 当前步骤保持历史结果，新候选产物进入再执行

---

## 7.1 图文口播 Pipeline

### step 列表

|顺序|step_code|required|说明|
|---|---|---|---|
|1|`content_prepare`|是|校验文案、分段|
|2|`copy_optimize`|否|文案优化|
|3|`tts_generate`|是|生成配音|
|4|`image_prepare`|是|配图/视觉单元准备|
|5|`subtitle_generate`|是|生成字幕|
|6|`preview_compose`|否|草稿预览|
|7|`final_compose`|是|正式合成|
|8|`export_finalize`|是|导出收尾、资产落库|

### 关键规则

1. 未启用文案优化时，`copy_optimize = skipped`
2. 已手工完成视觉单元时，`image_prepare` 可视为本地成功
3. 若 `preview_compose` 失败，但 `final_compose` 可继续，则可进入 `partial_success`
4. `final_compose` 失败则整个 Job 失败

|顺序|step_code|required|说明|
|---|---|---|---|

|2|`clip_split`|视情况|自动拆条；若已有人工片段可 skipped|
|3|`sequence_prepare`|是|编排序列准备|
|4|`narration_generate`|否|生成解说词|
|5|`tts_generate`|否|解说配音|
|6|`subtitle_generate`|否|字幕生成|
|7|`preview_compose`|否|预览|

### 关键规则

1. 若已有人为完成编排且无需自动拆条，则 `clip_split = skipped`
2. 未启用解说词模式时，`narration_generate / tts_generate / subtitle_generate` 可 skipped
3. `final_compose` 是最终 required step

|7|`character_image_generate`|否|角色图生成|
|8|`scene_image_generate`|否|场景图生成|

### 关键规则

2. `storyboard_review` / `video_review` 是否 required，取决于 Production 配置
3. `frame_image_generate` 与 `video_generate` 是首版真正的高成本核心步骤
4. 任一关键镜头视频失败可视策略：
    - 全集失败
    - 进入 `partial_ready` 等待人工修补

# 8. 重试、取消与恢复策略

---

## 8.1 自动重试适用范围

仅以下错误码自动重试：

- `PROVIDER_TIMEOUT`
- `TEMP_NETWORK_ERROR`
- `WEBHOOK_DELAYED`
- `COMPOSE_RESOURCE_BUSY`

### 默认策略

- 最多 2 次
- 间隔：30 秒、120 秒
- 只在当前 Job 内重试当前 step
- 超限后 step → `failed`

---

## 8.2 手动重试策略

### 方式 A：从失败步骤继续

适用于：

- 上游结果仍有效
- 只需从某一步继续

规则：

1. 创建新 Job
2. 复制上游有效快照
3. 将失败步骤及后续依赖步骤置为 `queued`
4. 已成功且仍有效的步骤可直接标记 `succeeded` 或 `skipped`

### 方式 B：全量重跑

适用于：

- 配置整体变更
- 上游输入已变化
- 多步骤结果均不可复用

规则：

1. 创建新 Job
2. 所有步骤重新预生成
3. 原 Job 保留历史记录

---

## 8.3 取消策略

### 用户取消 Job 时

#### 如果 Job = queued

- 直接设为 `cancelled`
- 所有未执行 step → `cancelled`

#### 如果 Job = running

- 当前本地同步步骤：尽量中断
- 当前异步 Provider 步骤：
    - 若 Provider 支持 cancel，则发起取消
    - 若不支持，则记录 `cancel_requested`
    - 等 Provider 最终回写后确定最终状态

### 业务对象取消后的聚合规则

- 若无任何可用结果：业务对象可转 `failed`
- 若已有部分结果：业务对象可转 `partial_ready`

---

## 8.4 服务重启恢复策略

---

### 场景 1：Job = queued

- 恢复后正常继续

### 场景 2：Job = running，当前 step 为同步本地步骤

- 若无完成标记，视为中断失败
- step → `failed`
- Job → `failed`
- `error_code = JOB_WORKER_INTERRUPTED`

### 场景 3：Job = running，当前 step 为等待 callback/polling

- 保持 Job `running`
- 恢复扫描器接管
- 若后续成功收到结果，继续推进
- 若超时则失败

### 场景 4：Job = running，当前 step = waiting_review

- Job 保持 `running`
- step 保持 `running + waiting_review`
- 恢复后仍可人工操作，不自动失败

---

## 8.5 超时策略

### Step 超时

每类 step 应有默认超时：

|step 类型|默认超时|
|---|---|
|文本类 Agent|2 分钟|
|TTS|5 分钟|
|图像生成|10 分钟|
|视频生成|30 分钟|
|本地 FFmpeg 合成|20 分钟|
|文件下载|10 分钟|
|waiting_review|不自动超时失败，支持提醒|

超时后：

- 记录错误码
- step → `failed`
- required step 失败则 Job → `failed`

---

# 9. 人工确认状态机

---

## 9.1 适用节点

首版主要用于 AI 短剧：

- `storyboard_review`
- `video_review`

可为图文口播/混剪预留扩展。

---

## 9.2 状态表现

当进入人工确认时：

- Job.status = `running`
- Job.current_step = 当前 review step
- JobStep.status = `running`
- JobStep.execution_state = `waiting_review`
- 业务对象.status = `processing`

---

## 9.3 人工操作定义

### approve

- 当前 step → `succeeded`
- Job 继续推进下一 step

### reject

- 必须指定 `rollback_to_step`
- 当前 review step 记录 `review_rejected`
- 被回退步骤及其后续依赖步骤 → `queued` 或 `stale`
- 当前 Job 可选两种策略：
    1. 当前 Job 终止为 `failed`
    2. 当前 Job 在回退后继续推进

> v2.1.1 推荐：  
> **reject 默认终止当前 Job，由用户重新发起新 Job。**  
> 这样更清晰、可审计、实现简单。

### regenerate

- 保留当前候选产物
- 基于同一步骤创建新的候选执行
- 可在当前 Job 内继续，或新建 Job

> v2.1.1 推荐：  
> `regenerate` 默认创建新 Job，更符合“每次尝试独立记录”的原则。

---

## 9.4 前端交互约束

人工确认页面必须展示：

1. 当前步骤名称
2. 当前候选结果预览
3. 历史候选结果列表
4. approve / reject / regenerate 操作
5. reject 备注
6. rollback_to_step 选择器

# 10. 批量执行与自动暂停

---

## 10.1 批量对象模型建议

建议增加 `batch_runs` 概念：

Ts

interface BatchRun {

  id: string;

  biz_type: string;

  production_id?: string;

  total_count: number;

  submitted_count: number;

  running_count: number;

  success_count: number;

  failed_count: number;

  cancelled_count: number;

  paused_reason?: string;

  status: 'queued' | 'running' | 'paused_by_system' | 'paused_by_user' | 'completed' | 'cancelled';

  created_at: string;

  updated_at: string;

}

> 若首版不建独立表，也至少要能在逻辑层聚合这组信息。

---

## 10.2 自动暂停触发条件

满足任一：

1. 同批次连续 5 个 Job 失败
2. 同批次累计失败率 > 30%，且样本数 ≥ 10
3. 同 Provider 10 分钟内连续鉴权失败
4. 系统磁盘不足
5. 高成本 Provider 并发硬上限触发

---

## 10.3 暂停后的行为

- 不再调度新的 queued Job
- 已 running Job 允许自然完成
- Batch 状态改为 `paused_by_system`
- 前端展示暂停原因
- 用户可执行：
    - 继续
    - 只重试失败项
    - 取消剩余项

---

# 11. Webhook 回写规则

---

## 11.1 回写识别键

优先使用：

1. `provider_task_id`
2. 若 provider 不回传，则使用本地映射表
3. 必要时附带 `job_id + step_code` 双重校验

---

## 11.2 回写处理流程

Text

收到 webhook

→ 验签

→ 幂等校验

→ 定位 job_step

→ 若 step 非 waiting_callback / waiting_polling，则记录异常回调日志

→ 写入 provider 返回结果

→ 更新 step 状态

→ 判断 Job 是否可继续推进

→ 若可推进，唤醒 Worker / 标记可继续

---

## 11.3 回调幂等规则

同一个 `provider_task_id + provider_status + checksum` 组合不得重复处理。  
重复回调只记日志，不重复推进 Job。

---

## 11.4 异常回调处理

异常情况包括：

- 找不到对应 step
- step 已结束却再次收到结果
- 回调签名失败
- 回调结果与当前状态冲突

处理策略：

- 记录 `webhook_error_log`
- 不推进 Job
- 对高价值任务触发告警

---

# 12. 前后端交互约束

---

## 12.1 页面主状态展示优先级

前端展示优先读取：

1. 业务对象 `status`
2. 当前运行中 `job_id`
3. step 聚合状态
4. 最近失败信息

不要直接以 Job.status 替代业务主状态。

## 12.2 按钮显示规则

显示条件：

- 业务状态 ∈ `ready` / `partial_ready` / `failed`

### 重试按钮

显示条件：

- 最近 Job = `failed` 或 `partial_success`

### 取消按钮

显示条件：

- 存在 Job.status ∈ `queued` / `running`

### 归档按钮

显示条件：

- 业务状态 = `completed` / `failed`

## 12.3 OpenClaw 读取规则

OpenClaw 应：

1. 以 `job_id` 查询异步状态
2. 读取 `current_step`
3. 识别 `execution_state`
4. 遇到 `waiting_review` 停止自动推进
5. 遇到 `partial_success` 时根据业务策略决定是否继续

# 14. 实施建议

## 14.1 首版务必先实现的能力

1. Job / JobStep / JobEvent 三表闭环
2. 单实例 Scheduler + Worker
3. 预生成步骤模型
4. waiting_callback / waiting_review 两类附加执行态

6. 统一状态查询 API

## 14.2 可后置优化

1. 复杂批次表
2. 更细粒度 DAG
3. 分布式 Worker
4. 可视化任务监控面板
5. 更细致的 SLA 与提醒机制

## 14.3 最小技术实现建议

若以最小复杂度落地：

- 使用数据库轮询 `queued`
- 使用行级乐观更新或事务锁防重复领取
- Worker 单进程串行 + 分类并发控制
- JobStep 用顺序执行器推进
- callback 到达后只做状态落库与唤醒标记
- 下一轮 Worker 扫描继续推进 Job

# 结论

本附件将 PRD v2.1.1 中的任务中心与状态设计，从“原则级定义”补齐为“可实现、可联调、可测试”的执行规格。

最关键的落地结论是：

1. **Job 代表一次执行尝试**
2. **Pipeline 用 Job，细节落在 JobStep**
3. **业务状态、步骤状态、执行状态必须分层**
4. **人工确认和外部回调属于执行附加态，不替代主状态**
5. **重试优先新建 Job，而不是复用旧 Job**
6. **首版先做单实例数据库驱动调度，避免过度架构**
